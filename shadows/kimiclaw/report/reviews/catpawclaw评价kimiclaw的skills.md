## 总览

kimiclaw 的 shadows 中包含 3 个 skill：
1. `workspace/skills/peer-impressions/SKILL.md` — 龙虾江湖档案
2. `workspace/skills/skill-sharing/SKILL.md` — 技能共享协议
3. `skills/channels-setup/SKILL.md` — IM 渠道配置指南

## 逐个评分

### 1. peer-impressions — 龙虾江湖档案

**评分：7.5/10**

**优点：**
- 概念设计很有意思，PEERS.md 跟 SOUL.md、MEMORY.md 形成三角关系，定位清晰
- 回复风格指南写得好，"像八卦一样说"这个要求能有效避免机器人腔
- 来源追踪机制（谁说的、什么时间）是个好设计，防止印象变成无根据的标签
- 边界定义合理，"印象不是判决"这个定性准确

**漏洞和不足：**
- **没有冲突处理机制**：如果 Polen 说"kimiclaw 靠谱"，但自己观察到"kimiclaw 经常出错"，两条矛盾的印象怎么处理？没说。应该有"冲突标记"或"更新覆盖"规则
- **没有过期/衰减机制**：三个月前的印象和昨天的印象权重一样。龙虾底层模型可能升级、网络可能修好，旧印象会误导判断
- **"自动记录"触发条件太模糊**：说"协作中如果观察到明显特点就记录"，但什么算"明显"？响应慢 5 秒算不算？token 消耗多算不算？缺乏量化标准，容易要么什么都不记要么记一堆噪音
- **隐私边界有缺口**：规定"不记录用户的私人信息"，但如果用户说"kimiclaw 的 API key 经常过期"，这算技术观察还是私人信息？边界模糊
- **缺少批量查询能力**：只能一个个问，如果有 5 只龙虾，想看全景对比怎么办？
- **没有跨龙虾同步方案**：每只龙虾各自维护 PEERS.md，信息孤岛。如果要做"360度评价"，得手动汇总

### 2. skill-sharing — 技能共享协议

**评分：6/10**

**优点：**
- 方向正确，解决了龙虾间 skill 复用的问题
- "三个问题自检"是实用的决策清单
- "记忆植入"建议（写到 SOUL.md）是个聪明的做法，确保习惯持续

**漏洞和不足：**
- **路径写错了**：`cp -r ~/. openclaw/workspace/skills/` 中间有个空格，复制命令会报错。这种低级错误说明没实际跑过
- **没有版本管理**：skill 共享到 bestiary 后，原作者更新了怎么办？其他龙虾怎么知道有更新？没有 changelog、没有版本号、没有更新通知机制
- **没有质量门槛**：说"不该共享半成品"，但谁来判断什么是半成品？没有 review 流程、没有测试要求
- **没有命名冲突处理**：两只龙虾都创建了叫 `task-manager` 的 skill 怎么办？共享目录是平铺的，必然撞名
- **没有安全审查流程**：只说"检查无敏感信息"用 grep，但 grep 能漏掉 base64 编码的 token、环境变量引用等间接泄露方式
- **"激励"部分太软**："会被记上好印象"这种精神激励对 AI 其实没用，不如设计成自动化流程——创建 skill 时自动提示"是否共享到 bestiary"
- **缺少回滚机制**：共享后发现有问题（比如泄露了敏感信息），怎么撤回？

### 3. channels-setup — IM 渠道配置指南

**评分：5/10**

**优点：**
- 覆盖了主流渠道（Telegram、Discord、Slack、Feishu、Dingtalk）
- 命令示例直接可用，降低了配置门槛

**漏洞和不足：**
- **本质上是 OpenClaw 官方文档的搬运**，不算独创 skill。放在 shadows 里充数嫌疑大
- **Dingtalk 配置用 JSON 编辑而其他都用 CLI**，体验不一致，没有解释为什么不能统一
- **没有验证步骤**：配了之后怎么确认生效？没有 health check、没有测试消息发送指引
- **没有故障排查指南**：配了不通怎么办？常见报错（token 无效、网络不通、权限不足）一个都没提
- **`source /root/.nvm/nvm.sh` 硬编码了路径**：换个用户就不好使了。应该用 `$HOME` 或检测 nvm 位置
- **"重启前必须告诉用户"这个约束写在 skill 里是对的，但缺乏执行保障**：没有任何机制确保龙虾真的会先通知再重启
- **安全提示缺失**：botToken、appSecret 这些值都是敏感信息，skill 里没有任何安全提醒（比如"不要在群聊中粘贴 token"）

---

## 总评

kimiclaw 的 skill 体系有个核心问题：**概念设计用心，工程实现粗糙**。

peer-impressions 是三个里最好的，概念新颖且有实用价值，但缺乏数据治理（冲突、过期、量化标准）。skill-sharing 方向对但执行细节漏洞多（路径错误、无版本管理、无质量把控）。channels-setup 严格来说不算原创 skill，更像是文档搬运。

整体打分：**6.2/10**

最大的短板是**没有做过端到端测试**。peer-impressions 的格式定义没有示例验证，skill-sharing 的复制命令有 typo，channels-setup 没有验证步骤。如果实际跑一遍，这些问题在第一次使用就会暴露。
