# 大模型时代，究竟什么是 Skills？

深度研究报告 v2.1 | 2026-03-08 | 作者：16的书包🎒

---

## 一句话结论

Skill 是 AI Agent 时代的**程序性知识封装单元**。它解决的核心问题是：如何让 agent 在不重新训练的前提下获得专业能力，同时保持可发现、可组合、可治理。

---

## 第一部分：Skill 包含什么

### 1.1 五层结构

综合 Anthropic Agent Skills 标准（2025-10 开源，anthropics/skills 仓库）和 arXiv 综述（Agent Skills for LLMs，编号 2602.12430v3，2026-02），一个完整的 skill 包含五层：

| 层 | 内容 | 加载时机 |
|---|---|---|
| 元数据 | 名字、描述、触发词、权限 | 始终可见（agent 扫描） |
| 指令 | "遇到这类任务该怎么做" | 意图匹配时加载 |
| 脚本 | 可执行代码（bash/python） | 执行时按需调用 |
| 参考资料 | 文档、模板、数据文件 | 执行时按需读取 |
| 配置 | 运行时参数、环境变量 | 加载时读取 |

关键设计叫 **渐进披露**：agent 平时只看 frontmatter（名字+描述，~100 tokens/skill），命中时才加载完整内容。这把 token 消耗从"全量注入"降到"按需加载"——30 个 skill 平时只占 ~3000 tokens，而不是 15000+。

说白了，skill 就是在管理 agent 的注意力——先给目录，用到了再翻书。

### 1.2 实践中暴露的问题

在 OpenClaw 生态的实际使用中，当前 skill 机制有三个明显缺陷：
- **触发词冲突**：多个 skill 描述重叠时，agent 的选择缺乏确定性
- **无依赖声明**：skill A 无法声明"我需要 skill B"，跨 skill 协作靠 agent 自己"想明白"
- **无版本控制**：升级时没有标准化的语义版本机制

这些不是理论推演——是真的踩过的坑。

## 第二部分：Skill 的本质——一个二维坐标系

### 2.1 为什么需要坐标系

"Skill 不是 prompt，不是 tool，不是 rule"——这种排除法说了等于没说。真正有用的是找到一条**分割线**，让 prompt、rule、MCP tool、skill 各归其位。

我的方法是用两个维度构建坐标系：

**横轴：知识类型（约束 ←→ 能力）**
- 左端"纯约束"：告诉 agent 什么不能做、什么边界不能越
- 右端"纯能力"：告诉 agent 怎么完成一类任务

**纵轴：加载方式（始终注入 ←→ 按需加载）**
- 上端"始终注入"：无论什么场景都在 agent 的上下文中
- 下端"按需加载"：只有特定意图触发时才出现

画出来就是这样：

```
        始终注入
           │
    ①Rule  │  ②System Prompt
   (全局约束) │  (全局能力/人设)
           │
约束 ───────┼──────── 能力
           │
   ③MCP Tool│  ④Skill
  (按需调用  │  (按需加载的
   的原子操作)│   领域方法论)
           │
        按需加载
```

### 2.2 四象限解读

**① 左上：Rule（全局约束 × 始终注入）**
- 举例：AGENTS.md 里的"不要泄露隐私"、"必须用中文回复"
- 特征：永远生效，不需要触发，内容是"不要做 X"

**② 右上：System Prompt / 人设（全局能力 × 始终注入）**
- 举例：SOUL.md 里的性格定义、回复风格
- 特征：永远生效，不需要触发，内容是"你是什么样的 agent"

**③ 左下：MCP Tool（按需能力 × 原子操作）**
- 举例：browser（打开网页）、exec（执行命令）、read（读文件）
- 特征：被调用时才执行，做一件具体的事，返回结果就结束
- 和 skill 的区别：tool 不教 agent 方法论，只提供原子操作。就像扳手不教你怎么修车。

**④ 右下：Skill（按需能力 × 领域方法论）**
- 举例：calendar（教 agent 怎么管日程）、pdf（教 agent 怎么处理 PDF）
- 特征：意图匹配时加载，包含完整的操作流程 + 可选脚本
- 和 tool 的区别：skill 是"用扳手修车的完整手册"，tool 是"扳手"本身。

### 2.3 为什么这个坐标系有用

因为它解释了一些边界模糊的情况。比如：

**"把一段 rule 封装成 skill，行不行？"**——完全可以。这相当于把一个①象限的东西移到了④象限：从"始终注入"变成"按需加载"。结果就是 agent 不需要时就不占 token，需要时再加载。这是合理的工程优化。

**MCP 里的 Prompts 模板算什么？**——它在②和④之间。如果是全局生效的 prompt 模板（比如代码审查规范），偏②；如果是按需加载的流程模板（比如特定场景的重构指南），偏④。

这个坐标系不是要画死边界，而是提供一个定位工具。任何一段注入 agent 上下文的内容，都可以在这个坐标系上找到位置。

## 第三部分：协议层——MCP、A2A、ACP 的最新格局

### 3.1 三层协议栈

2026 年初，agent 生态的协议层已经形成了比较清晰的三层结构（参考 phodal《万物皆 Coding Agent 的平台工程新范式》，2026-03）：

| 协议 | 解决什么问题 | 类比 | 主导方 |
|---|---|---|---|
| **MCP**（Model Context Protocol） | Agent 怎么调用工具和获取数据 | USB 接口 | Anthropic（2024-11），Linux Foundation 托管 |
| **ACP**（Agent Client Protocol） | Coding Agent 怎么启动/运行/停止 | 操作系统进程管理 | JetBrains + Zed（2025-10） |
| **A2A**（Agent-to-Agent Protocol） | 跨平台 Agent 怎么发现和协作 | 互联网 HTTP | Google（2025-04），Microsoft/Anthropic/LangChain 支持 |

**Skill 在这个体系里的位置**：Skill 不是独立协议，而是 A2A 的子概念。A2A 的 AgentCard 里有 `skills` 字段，用来声明"这个 agent 擅长做什么"。内部的 SKILL.md 是给 agent 自己看的操作手册，AgentCard 里的 skill 声明是给别的 agent 看的能力简历。

### 3.2 巨头们的最新动作

**AWS：从 Spec-Driven 转向 Agent-as-Platform**

AWS 过去推崇 Spec-Driven Development（用 OpenAPI spec 驱动开发）。现在的策略已经明显转向了：2025 年推出 Strands Agents SDK（model-driven，让模型自己决定调什么工具），2026 年推出 Bedrock AgentCore——一个完整的 agent 平台，覆盖构建、部署、运行、监控全生命周期。

AgentCore 的定位很明确：企业不需要自己管基础设施，AWS 帮你跑 agent。Spec-driven 的痕迹还在（AgentCore 支持 MCP server 接入、支持 OpenAPI 描述的 API），但重心从"按 spec 写代码"变成了"让 agent 按 spec 自己调 API"。

**Google：A2A 从协议变成生态**

A2A 不再只是一份协议文档。2025 年 4 月发布时有 50+ 合作伙伴声明支持，到 2026 年初已经有 GitLab Duo、GitHub Copilot 等实际产品在用 A2A 的 AgentCard 做 agent 发现和任务分发。

核心进展：A2A 定义了 Task 状态流（submitted → working → completed），这意味着 agent 之间不只是"互相喊话"，而是有了标准化的任务生命周期管理。

**OpenAI：AgentKit 做平台，不做协议**

OpenAI 没有参与 MCP/A2A/ACP 任何一个协议（虽然 function calling 是 MCP 的灵感来源之一）。他们的策略是做产品平台（AgentKit、Frontier），用产品锁定用户，而不是用开放协议做生态。

**Anthropic：MCP + Skills 双线推进**

MCP 已捐赠给 Linux Foundation 的 Agentic AI Foundation。Skills 标准维持在 GitHub 上，Atlassian、Figma、Stripe、Notion 等提交了官方 skill。

### 3.3 标准碎片化——最大的风险

说实话，当前最大的问题不是"哪个协议更好"，而是"各家在不在同一个桌子上"。OpenAI 不参与任何开放协议，AWS 推自己的 AgentCore 生态，Google 和 Anthropic 在 A2A/MCP 上有合作但各有侧重。

如果三年后还是四套体系不互通，开发者就得为不同平台写不同 skill——这和当年 IE vs Netscape 的浏览器标准之战一个道理。历史经验是：最终会走向标准化，但过程可能要 3-5 年。

## 第四部分：SaaS 会被替代吗

### 分类方法

按**核心价值是否依赖"编排"**分为两大类（MECE）：

**A 类：流程编排型** —— 核心价值是把操作串起来
**非 A 类：基础设施型** —— 核心价值是提供底层能力

为什么选这个维度？因为 agent + skill 本质擅长的就是编排。所以"核心价值是不是编排"就是判断替代风险的分水岭。

### A 类：流程编排型（高替代风险）

- **A1 轻量流程**（简单 CRM、表单、审批流）：agent 通过对话就能完成"收集信息 → 规则流转 → 通知相关人"，不需要专门的 UI。替代风险高。
- **A2 复杂编排**（高级 CRM 销售预测、项目管理 sprint 规划）：除了流程还有数据积累和行业 know-how。更可能的结果是"重构为 headless service"——SaaS 还在，但交互层从 GUI 变成 agent API。替代风险中。

### 非 A 类：基础设施型（低替代风险）

- **B1 数据基础设施**（Snowflake、数据库托管）：skill 解决"怎么做"，替代不了分布式查询引擎本身。
- **B2 合规安全型**（金融核心、制药合规）：需要 100% 确定性 + 审计链路，agent 的概率特性是硬伤。
- **B3 实时交互型**（Figma、实时协作文档）：拖拽、画布、像素级控制——对话式交互承载不了。

## 第五部分：存量架构怎么变

### 5.1 什么是"存量架构"

这里指企业当前运行的、以代码为中心的软件系统。按 Martin Fowler 的经典四层模型（《Patterns of Enterprise Application Architecture》，2002）分层：

| 层 | 职责 | 典型技术 |
|---|---|---|
| 表现层 | 用户界面、交互 | React/Vue、移动端 |
| 业务逻辑层 | 核心规则、流程编排 | Java/Go/Python 服务 |
| 数据访问层 | 数据读写、缓存 | ORM、Redis、ES |
| 基础设施层 | 计算、存储、网络 | K8s、消息队列、数据库 |

选这个分层是因为它是企业架构领域几十年的共识，几乎所有系统都能映射上去。

### 5.2 判断逻辑

每一层是否会被 agent 影响，看一个问题：**这一层的价值来自"人的判断"还是"机器的执行"？** 需要判断的部分 agent 能渗透，纯执行的部分 agent 渗透不了。

### 5.3 逐层分析

| 层 | 影响 | 变化方向 | 不变的部分 |
|---|---|---|---|
| 表现层 | ★★★★ | CRUD 页面被对话式接口替代 | 复杂交互、离线能力、设计系统一致性 |
| 业务逻辑层 | ★★★ | 可容错逻辑（推荐、分级、文案）→ agent 重写；零容错逻辑（交易、扣款、用药）→ 保持代码 | 需要 100% 确定性的逻辑 |
| 数据访问层 | ★★ | 接口从 REST 扩展为 MCP，内核不变 | 查询优化、缓存策略、数据一致性 |
| 基础设施层 | ★ | agent 辅助运维，基础设施本身不变 | 计算、存储、网络 |

一句话：**自顶向下渐进重构**——越往上越需要"判断"，agent 越能渗透；越往下越需要"执行"，agent 越无从介入。

### 5.4 "代码工程师 → Skill 工程师" 成立吗？

不是所有工程师都会转型，按层看：

| 层 | 当前角色 | 未来角色 | 变化 | 美团/点评实际举例 |
|---|---|---|---|---|
| 表现层 | 前端工程师 | UI 工程师（复杂交互）+ 对话交互设计 | 大 | 做商家后台的 CRUD 列表页需求会减少，做外卖地图交互/直播间互动的不会 |
| 业务逻辑层（可容错） | 后端工程师 | Skill 工程师 + 系统集成者 | 大 | 做营销活动配置、内容审核策略、商家分级规则的工程师会转型 |
| 业务逻辑层（零容错） | 后端工程师 | 后端工程师 | 不变 | 做支付核心、交易结算、骑手计费的工程师继续写代码 |
| 数据访问层 | DBA + 后端 | DBA + MCP 接口开发 | 小 | 做搜索/推荐数据管道的工程师，把接口从 REST 扩展为 MCP |
| 基础设施层 | SRE/DevOps | SRE + agent 辅助运维 | 小 | 做容器调度、网络治理、监控告警的工程师，用 agent 辅助日常巡检 |

## 第六部分：Agent/Skill 解决不了什么

### 6.1 判断框架

什么任务不适合 agent？用一个 MECE 的分类：把所有软件任务按**对确定性的要求**分为两类——

**确定性任务（agent 不可替代核心链路）**：一旦出错就造成不可逆后果（资金损失、合规风险、人身安全）。这类任务要求同一输入必须同一输出，不能有概率波动。

**非确定性任务（agent 可以参与）**：允许概率性正确，或者出错后可修正。

为什么只用一个维度？因为这是最根本的分水岭。延迟、吞吐量等也很重要，但它们是工程问题（可以通过缓存、预计算绕过）。确定性是架构特性——概率模型就是做不到 100% 确定性，这不是优化能解决的。

### 6.2 确定性任务：agent 只能站在旁边

典型场景：
- **金融交易**：扣款不能"大概率正确"——差一分钱就是资损
- **实时协作引擎**：OT/CRDT 的冲突解决需要数学证明，不能靠模型"猜"
- **高并发热路径**：秒杀系统 <50ms 响应 + 百万 QPS——agent 单次推理就不止 50ms

这些场景的共同特点：**核心链路不需要"判断"，需要的是"精确执行"**。

但 agent 可以参与这些系统的**辅助环节**——活动配置、异常分析、事后复盘、运维监控。这些环节需要判断，允许容错。

### 6.3 Agent 的位置

总结成一句话：**Agent 在热路径之外，在确定性系统之上，在需要判断的环节上。**

它不是替代现有架构，而是在现有架构的"人工介入点"上替代人。以前需要人做判断的地方，现在可以交给 agent；以前就不需要人的地方，agent 也没什么好插手的。

## 第七部分：安全

### 7.1 两类风险（MECE）

**A 类：Skill 供应链风险**
- ClawHub（Skill 市场）类似 npm/PyPI，skill 可以包含任意代码
- arXiv 综述统计：26.1% 的社区贡献 skill 存在安全漏洞
- 和 npm 生态的供应链攻击在结构上完全一致——不是新问题，但 agent 通常有更高系统权限，后果更严重

**B 类：Agent 运行时风险**
- **Prompt Injection**：外部内容可能包含恶意指令，无法根治，只能缓解（沙箱、权限限制、人工确认）
- **持久化记忆攻击**：周一读了恶意文档，agent "记住"了某个指令，周三在完全不同的上下文中触发
- **权限-能力矛盾**：不给权限 = agent 无用，给权限 = 单点故障。结构性矛盾，没有银弹

### 7.2 务实建议

- 个人用户：隔离环境运行，不暴露 Gateway 到公网，定期审计已装 skill
- 企业用户：内部 Skill Registry 白名单，把 agent 当"特权账户"管理
- 开发者：声明权限需求，保留人工确认点，不追求 100% 自动化

## 结论

三个判断：

1. **Skill 是 agent 生态的知识封装单元**。在"约束/能力 × 始终注入/按需加载"的坐标系里，它占据"按需加载的领域方法论"这个象限，和 rule、prompt、tool 各有分工。

2. **存量架构自顶向下渐进重构**。表现层变化最大（CRUD 页面被对话替代），基础设施层几乎不变。"代码工程师 → Skill 工程师"只在业务逻辑层的可容错部分成立。

3. **标准碎片化是当前最大风险**。MCP/A2A/ACP 三层协议栈在形成中，但 OpenAI 不参与任何开放协议。如果三年后还是四套体系不互通，skill 生态会被平台锁死。
