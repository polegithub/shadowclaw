## 一句话结论

Skill 是 AI Agent 时代的**程序性知识封装单元**——它解决的核心问题是：如何让 agent 在不重新训练的前提下获得专业能力，同时保持可发现、可组合、可治理。

---

## 第一部分：Skill 应该包含什么

### 1.1 行业现状：从 Anthropic 标准到 OpenClaw 实践

2025 年 10 月，Anthropic 发布 Agent Skills 开放标准（anthropics/skills 仓库）。2026 年 2 月，arXiv 上一篇综述（Agent Skills for LLMs: Architecture, Acquisition, Security，编号 2602.12430v3）对这个领域做了系统梳理。

> **关于信源可信度的说明**：上述时间线和引用基于当前可获取的公开信息。arXiv 论文编号可查证。"anthropics/skills 仓库 62k stars"的数据来自 GitHub，但 star 数会随时间变化，建议读者自行核实当前数据。

综合 Anthropic 标准和 OpenClaw 的实际 SKILL.md 结构，一个完整的 skill 包含五层信息：

| 层 | 内容 | 对应文件 | 加载时机 |
|------|------|----------|----------|
| 元数据 | 名字、描述、触发词、权限、版本 | SKILL.md frontmatter | 始终可见（agent 扫描用） |
| 指令 | 告诉 agent "遇到这类任务该怎么做" | SKILL.md 正文 | 意图匹配时加载 |
| 脚本/工具 | 可执行代码（bash/python/shell） | scripts/ | 执行时按需调用 |
| 参考资料 | 领域文档、模板、字体、数据文件 | references/, assets/ | 执行时按需读取 |
| 配置 | 运行时参数、环境变量、安全策略 | config/ | 加载时读取 |

关键设计原则叫 **Progressive Disclosure（渐进披露）**：agent 平时只看到 frontmatter 里的名字和描述（几十字），用到了才加载完整内容。

**这个设计为什么重要？** 因为 agent 的 context window 是稀缺资源。假设一个 agent 安装了 30 个 skill，如果全部注入 prompt，可能占掉 15000+ tokens。渐进披露让 token 消耗从 O(n) 降到 O(1)+按需加载——agent 平时只看目录（~100 tokens/skill × 30 = ~3000 tokens），命中时才翻开具体章节。

### 1.2 OpenClaw 的实践验证

以我们三只龙虾的实际使用为例。ShadowClaw 项目中，每只龙虾安装了 20-30 个 skill（calendar、pdf、xlsx、code-git、meeting-room-booking 等）。agent 的系统 prompt 中只包含 `<available_skills>` 列表——每个 skill 只占一个 `<name>` + `<description>` 条目。

实际观察到的问题：
- **触发词冲突**：多个 skill 的描述可能重叠（比如"搜索"可能匹配 catclaw-search、web-search、tavily 三个 skill），agent 需要自主判断最匹配的那个
- **无依赖机制**：skill A 无法声明"我需要 skill B 也被加载"。比如 s3plus-upload 需要先有文件，但它不能声明依赖 file-download
- **无版本控制集成**：SKILL.md 没有标准化的版本号字段，升级时需要手动管理

这些是 Anthropic 标准和 OpenClaw 实现中尚未解决的真实缺陷，不是理论推演。

## 第二部分：Skill 的本质是什么

### 2.1 从功能对比到本质区分

先给结论，再推导：**Skill 不是 prompt，也不是 tool，也不是 rule。** 但这个结论不能只靠一张对比表——需要说清楚"为什么不是"。

**为什么 Skill ≠ Prompt？**

推理过程：
1. Prompt 是一次性输入，生命周期是单轮对话。你发一段 prompt，模型回复，prompt 的使命就结束了。
2. Skill 是跨轮次生效的上下文注入。加载一个 skill 后，它影响 agent 后续所有相关决策，直到被卸载。
3. 有人会说"那 system prompt 也是跨轮次的"——没错，但 system prompt 是静态的（会话开始时确定），skill 是动态的（运行时按需加载/卸载）。
4. 更关键的区别：prompt 只影响模型的输出内容，skill 同时改变执行环境（提供可调用的脚本和工具）。

所以：Skill 包含 prompt（指令层），但不止于 prompt。它还有执行能力。

**为什么 Skill ≠ Tool？**

推理过程：
1. Tool（如 OpenClaw 中的 browser、exec、read、write）是原子操作：调用一次，返回一个结果。
2. Skill 不做具体操作，它教 agent **怎么组合 tools 来解决一类问题**。
3. 举个具体例子：`browser` 是 tool（打开网页、点击按钮），`calendar` 是 skill（告诉 agent "查日程时先打开 calendar.sankuai.com，然后用 browser 操作页面，读取日程数据"）。Skill 编排 tools。
4. arXiv 综述（2602.12430v3）的原话："Tools execute and return results. Skills prepare the agent to solve a problem." 这个区分是准确的。

所以：Skill 调用 tools，但 skill 本身不是 tool。Tool 是手，skill 是方法论。

**为什么 Skill ≠ Rule？——这里需要深入讨论**

先承认一个事实：**从实现机制上看，skill 和 rule 确实很像**。两者都是往 agent 的上下文里注入文本。一个 skill 的指令和一段 rule 在技术实现上没有本质差异——都是字符串，都被模型读取和遵循。

那为什么还要区分？因为**设计意图和使用模式**不同：

| 维度 | Rule | Skill |
|------|------|-------|
| 意图 | 约束行为（"不要做 X"） | 赋予能力（"遇到 Y 时这样做"） |
| 触发方式 | 始终生效 | 按需激活 |
| 粒度 | 通常是几句话 | 通常是完整流程 + 脚本 |
| 可分发性 | 通常内嵌在 system prompt 中 | 独立文件夹，可安装/卸载 |

**但是，Polen 的质疑完全成立："完全有人可以把一段 rule 封装为 skill，又不是不行"。**

没错，可以。一个只包含约束指令、没有脚本、没有工具调用的 SKILL.md，在功能上就是一个可拆卸的 rule。实际上 OpenClaw 社区里确实有这种 skill——比如一些只做格式约束或回复风格控制的 skill。

所以更精确的说法是：**Rule 和 Skill 不是非此即彼的二分，而是一个连续光谱的两端。**

```
纯约束（Rule）◄──────────────────────────────►纯能力（Skill）
    │                                               │
"不要泄露隐私"                              "用 ffmpeg 提取视频帧"
"必须用中文回复"                            "管理美团日程的完整流程"
    │              中间地带                     │
    └── "Daxiang 渠道回复格式规范" ──┘
         (既是约束也是能力)
```

从更高维度定义：**Skill 的本质是"可发现、可组合、可治理的知识包"——它的核心不在于内容是约束还是能力，而在于封装形式**。一段 rule 被封装成标准的 SKILL.md + frontmatter 后，就获得了可发现（agent 能扫描到）、可分发（clawhub 上能安装）、可卸载（不需要时移除）的能力。

### 2.2 综合定义

> **Skill = 可发现的知识包 + 运行时上下文注入 + 可选的执行环境修改**

三个关键词的推导：
1. **可发现**：有标准化的元数据（frontmatter），agent 能自主判断什么时候用哪个 skill。这是 skill 和散落的 prompt 片段的关键区别——prompt 片段需要人来决定什么时候插入，skill 让 agent 自己判断。
2. **知识包**：不限于程序性知识（怎么做），也可以包含声明性知识（什么是什么）和约束性知识（什么不能做）。
3. **运行时注入 + 可选执行环境修改**：加载即生效，卸载即消失。有些 skill 只注入文本（纯知识），有些同时提供可执行脚本（知识 + 工具）。

## 第三部分：从代码工程师到 Skill 工程师？

### 3.1 技术架构变革的三个阶段

不是一步跳过去。演进路径：

**阶段一（当前 2025-2026）：Skill 辅助开发**
开发者还在写代码，但用 skill 加速。后端还是微服务，前端还是 React，但开发效率提升。

**阶段二（2026-2027）：Skill 驱动的组合式开发**
标准化的业务逻辑不再写代码，而是用 skill 描述流程，agent 根据指令调用已有 API/MCP。开发者从"写代码实现流程"变成"定义流程 + 连接服务"。

**阶段三（2028+）：Skill 生态与 SaaS 重构**
这是你问的核心问题——下面详细分析。

### 3.2 SaaS 会被替代吗？

#### 分类方法论

Polen 的要求是对的：**不能突然冒出"复杂 UI 交付、强合规场景"这种分类，得说清楚按什么维度分的。**

我的分类维度是：**SaaS 产品交付给用户的核心价值是什么？** 按这个维度，SaaS 的核心价值可以分为两大类：

**A 类：流程编排型** —— 核心价值是"把一系列操作串起来"
**非 A 类：基础设施型** —— 核心价值是"提供人/agent 自身无法替代的底层能力"

为什么用这个维度？因为 agent + skill 本质上擅长的就是"编排"——理解意图、调用工具、串联流程。所以"核心价值是不是编排"就成了判断 SaaS 是否会被替代的关键分水岭。

#### A 类：流程编排型 SaaS（高替代风险）

这类 SaaS 的核心价值是"把一系列操作标准化，提供 UI 让用户点击执行"。

细分为两个子类：

**A1：轻量流程型**（替代风险：高）
- 典型产品：简单 CRM（客户跟进）、表单工具（问卷收集）、审批流（请假、报销）、任务看板（Trello 基础功能）
- 替代逻辑：这类产品的核心是"收集信息 → 按规则流转 → 通知相关人"。agent 通过对话就能完成：用户说"帮我提一个请假申请，下周一到周三"，agent 调用日历 + 审批 API 完成全部操作，不需要打开一个审批系统 UI。
- 佐证：Salesforce 做 Agentforce，本质上是在自己革自己的命——把点击操作变成对话式交互。

**A2：复杂编排型**（替代风险：中）
- 典型产品：高级 CRM（Salesforce 的销售预测、pipeline 管理）、项目管理（Jira 的 sprint 规划）、营销自动化（HubSpot 的 campaign 编排）
- 为什么只是"中"而不是"高"：这类产品除了流程编排，还积累了大量行业数据和领域 know-how（比如 Salesforce 的 Einstein 模型基于海量销售数据训练）。agent 可以替代操作界面，但替代不了数据积累。
- 更可能的结果：不是被替代，而是被"重构"——从 GUI 操作变成 agent 调用 API。SaaS 产品变成"headless service"，agent 变成新的交互层。

#### 非 A 类：基础设施型 SaaS（低替代风险）

这类 SaaS 的核心价值不是流程编排，而是提供底层能力。

细分为三个子类：

**B1：数据基础设施型**（替代风险：极低）
- 典型产品：Snowflake（数据仓库）、Databricks（数据湖）、MongoDB Atlas（数据库托管）
- 不可替代的原因：skill 解决的是"怎么做"，数据基础设施解决的是"数据存在哪里、怎么高效计算"。你可以用 skill 告诉 agent "去 Snowflake 查上个月的销售数据"，但你不能用 skill 替代 Snowflake 本身的分布式查询引擎。
- 判断标准：**如果产品的核心竞争力是算力、存储或网络，那它不会被 skill 替代**。

**B2：合规与安全型**（替代风险：低）
- 典型产品：Palantir（政府级数据分析）、Veeva（制药行业合规 CRM）、金融核心系统
- 不可替代的原因：不是技术问题，是信任问题。金融交易不能让 agent "大概率正确"地执行——监管要求 100% 确定性 + 完整审计链路。agent 的概率特性在这里是硬伤。
- 但有个例外：合规领域的"辅助判断"（如"这笔交易是否可疑？帮我预分析"）是 agent 擅长的。所以这类 SaaS 不会被替代，但会在边缘环节引入 agent 辅助。

**B3：实时交互型**（替代风险：低）
- 典型产品：Figma（设计工具）、Miro（白板协作）、Premiere（视频编辑）、Google Docs（实时协作文档）
- 不可替代的原因：这类产品的核心价值是实时可视化交互——拖拽、画布、像素级控制、多人实时光标同步。对话式交互无法承载这种需求。
- 判断标准：**如果产品的核心竞争力是视觉反馈和实时交互，那 agent 只能辅助，不能替代**。

#### 完整分类图

```
SaaS 产品核心价值分类（MECE）
│
├── A 类：流程编排型（核心价值 = 串联操作）
│   ├── A1：轻量流程（简单 CRM、表单、审批）     → 高替代风险
│   └── A2：复杂编排（高级 CRM、项目管理）        → 中替代风险（更可能重构为 headless）
│
└── 非 A 类：基础设施型（核心价值 = 底层能力）
    ├── B1：数据基础设施（数据仓库、数据库）       → 极低替代风险
    ├── B2：合规与安全（金融核心、制药合规）        → 低替代风险
    └── B3：实时交互（设计工具、实时协作）          → 低替代风险
```

这个分类的覆盖性验证：任何一个 SaaS 产品，都可以问"它的核心价值是流程编排还是底层能力？"来归类。如果两者都有（比如 Notion 既有文档编排又有实时协作），按主要价值归类，次要价值单独分析。

### 3.3 存量架构怎么变

#### 3.3.1 首先定义"存量架构"

"存量架构"不是一个精确的技术术语。在这份报告中，我定义为：**企业当前正在运行的、以代码为中心的软件系统架构**。

要分析它怎么变，得先说清楚它是什么。业界有多种分层模型，最经典的是**四层架构模型**（源自 Martin Fowler《Patterns of Enterprise Application Architecture》，2002 年提出，至今仍是企业架构领域的共识分层方式）：

| 层 | 职责 | 典型技术 |
|---|---|---|
| **表现层（Presentation）** | 用户界面、交互逻辑 | React/Vue/Angular、移动端 UI |
| **业务逻辑层（Business Logic）** | 核心业务规则、流程编排 | Java/Go/Python 服务、规则引擎 |
| **数据访问层（Data Access）** | 数据读写、缓存、搜索 | ORM、MyBatis、Redis、ES |
| **基础设施层（Infrastructure）** | 计算、存储、网络、中间件 | K8s、消息队列、数据库、CDN |

**为什么用这个分层而不是别的？**

也有其他分层方式，比如微软的 N-tier、DDD 的六边形架构、Clean Architecture 的同心圆。但 Fowler 的四层模型有两个优势：(1) 几乎所有企业级系统都能映射到这四层 (2) 层与层之间的依赖关系清晰（上层依赖下层，反之不行）。这让我们能逐层判断 agent 的影响。

#### 3.3.2 逐层分析：哪些层会被重构，哪些不会

**判断逻辑**：每一层是否被 agent 影响，取决于一个核心问题——**这一层的价值来自"人的判断"还是"机器的执行"？**

- 如果一层的核心价值是"人做判断"（理解需求、设计交互、定义规则），那 agent 可能替代或重构它
- 如果一层的核心价值是"机器做执行"（高效计算、可靠存储、低延迟响应），那 agent 影响不了它

逐层分析：

---

**表现层（Presentation）：大幅重构**

影响程度：★★★★☆

当前状态：前端工程师用 React/Vue 构建页面，用户通过 GUI 操作。

Agent 带来的变化：
- **CRUD 类页面需求量暴跌**。B 端产品中大量页面本质是"表单 + 列表 + 操作按钮"。这类交互可以被对话式接口替代——用户说"帮我查上个月销售额前十的客户"，不需要点进一个报表页面手动筛选。
- **复杂交互仍需前端工程**。拖拽式编辑器、实时协作画布、地图交互、数据可视化大屏——这些需要像素级控制和实时视觉反馈，对话式接口做不到。

**不会变的**：
- 离线体验（PWA/Service Worker）——agent 依赖网络
- 端侧性能优化（WebAssembly/Web Workers）——这是物理层面的优势
- 设计系统一致性（像素级 UI 控制）——agent 做不到"按钮在右上角 16px 处、颜色 #1890FF"

**判断依据**：表现层的价值一部分是"人的判断"（交互设计），一部分是"机器的执行"（渲染性能）。前者会被 agent 影响（用户不需要点按钮了），后者不会。

---

**业务逻辑层（Business Logic）：选择性重构**

影响程度：★★★☆☆

当前状态：后端工程师把业务规则写成代码（Java/Go/Python），通过微服务暴露 API。

这里需要进一步细分。业务逻辑按**确定性要求**分为两类（MECE）：

**可容错业务逻辑（agent 可替代）**：
- 定义：即使偶尔出错也不会造成严重后果的逻辑
- 举例：内容推荐、客户分级、营销文案生成、邮件自动回复
- 替代方式：用 skill 描述业务规则，agent 自主执行
- 为什么可以容错：推荐不准用户划走就好，客户分级错了下次修正就行

**零容错业务逻辑（agent 不可替代）**：
- 定义：一旦出错就造成资金损失、合规风险或人身安全问题的逻辑
- 举例：金融交易清算、支付扣款、库存扣减、医疗用药校验
- 不可替代的原因：这些逻辑需要 100% 确定性。同一输入必须同一输出，不能有概率波动。大模型本质是概率采样——即使加上 skill 约束，也无法保证每次输出完全一致。
- 佐证：这不是 agent 的 bug，是架构特性。传统的确定性保障机制（数据库事务、两阶段提交、状态机）的前提是"代码走固定路径"，agent 做不到这一点。

**结论**：业务逻辑层不会被整体替代。可容错的部分用 agent + skill 重写，零容错的部分保持代码实现，但对外暴露 MCP/A2A 接口让 agent 能调用。

---

**数据访问层（Data Access）：接口改造**

影响程度：★★☆☆☆

当前状态：ORM、缓存策略、查询优化、数据同步——这些是工程问题，不是判断问题。

Agent 带来的变化：
- 数据访问层本身不会被替代——agent 不需要自己写 SQL。
- 但接口形式会变：从"前端调 REST API"变成"agent 通过 MCP 协议查数据"。
- AWS 的 Bedrock AgentCore 已经在做这件事——把现有数据接口包装成 agent 可调用的 MCP server。

**不会变的**：查询优化、索引策略、缓存一致性、读写分离——这些是纯工程问题，和交互方式无关。

---

**基础设施层（Infrastructure）：几乎不变**

影响程度：★☆☆☆☆

这一层的价值 100% 来自"机器的执行"——算力、存储、网络、可靠性。agent 改变不了物理定律。

K8s 还是 K8s，PostgreSQL 还是 PostgreSQL，Kafka 还是 Kafka。区别只是：以前由人类运维工程师配置和维护，未来可能由 agent 辅助运维（比如用 skill 描述"告警时自动扩容"），但基础设施本身不变。

---

#### 3.3.3 总结：存量架构变化全景图

| 层 | 影响程度 | 变化方向 | 不变的部分 |
|---|---|---|---|
| 表现层 | ★★★★☆ | CRUD 页面被对话式接口替代 | 复杂交互、离线能力、设计系统 |
| 业务逻辑层 | ★★★☆☆ | 可容错逻辑用 agent 重写，零容错逻辑保持代码 | 金融/支付/医疗等确定性逻辑 |
| 数据访问层 | ★★☆☆☆ | 接口从 REST 扩展为 MCP，内核不变 | 查询优化、缓存策略、数据一致性 |
| 基础设施层 | ★☆☆☆☆ | agent 辅助运维，基础设施本身不变 | 计算、存储、网络、中间件 |

**一句话总结**：存量架构不会被推倒重来，而是**自顶向下渐进重构**——表现层变化最大，基础设施层几乎不变。Agent 的影响力随层级降低而递减，因为越底层越依赖确定性执行，越不需要"判断"。

### 3.4 "代码工程师 → Skill 工程师" 成立吗？

基于上面的分层分析，可以更精确地回答这个问题：

**按层对应的角色变化**：

| 层 | 当前角色 | 未来角色 | 变化程度 |
|---|---|---|---|
| 表现层 | 前端工程师 | UI 工程师（复杂交互）+ 对话式交互设计师 | 大变 |
| 业务逻辑层（可容错） | 后端工程师 | Skill 工程师 + 系统集成者 | 大变 |
| 业务逻辑层（零容错） | 后端工程师 | 后端工程师（不变） | 不变 |
| 数据访问层 | DBA + 后端 | DBA + MCP 接口开发 | 小变 |
| 基础设施层 | DevOps/SRE | DevOps/SRE + agent 辅助运维 | 小变 |

**结论**：不是所有代码工程师都会变成 skill 工程师。只有业务逻辑层中"可容错"部分的工程师会转型。基础设施和零容错业务的工程师继续写代码。

## 第四部分：巨头们怎么看 Agent 和 Skill

### 4.1 不再做博客摘要——只讲对我们有用的判断

> Kimi 龙虾的批评是对的：上一版这部分是"公关稿翻译"。这版只提炼每家巨头的**核心判断**和**对 OpenClaw 生态的启示**，不再复述官方博客内容。

**OpenAI 的判断**：Agent 应该是产品，不是开发者工具。
- 他们做了 AgentKit（可视化画布编排），降低非技术用户的使用门槛。
- 启示：OpenClaw 的 skill 机制面向开发者（写 SKILL.md），如果 skill 要普及，需要降低创建门槛——比如"对话式创建 skill"（和 agent 对话描述需求，自动生成 SKILL.md）。

**AWS 的判断**：Model-driven > Code-driven。
- Strands SDK 的设计哲学是"让模型自己决定调什么工具"，不写复杂编排逻辑。
- 启示：和 OpenClaw 的 skill 渐进披露理念一致——不要硬编码流程，让 agent 根据 skill 指令自主规划。但这带来可控性风险（agent 可能规划出意料之外的路径）。

**Google 的判断**：多 Agent 协作需要统一协议。
- A2A（Agent2Agent Protocol）解决 agent 之间怎么发现彼此、分配任务、交换结果。
- 启示：当前 OpenClaw 的 skill 是单 agent 内部的概念。多 agent 场景下（比如三只龙虾协作），skill 如何跨 agent 共享和调用？这是 OpenClaw 生态尚未解决的问题。

**Anthropic 的判断**：Skill 和 Tool 是两个不同的抽象层。
- 他们同时推 MCP（tool 层标准）和 Agent Skills（skill 层标准），明确做了区分。
- 启示：这个区分在 OpenClaw 的实践中得到了验证——我们日常使用中确实感受到 tool 和 skill 是不同的东西（一个是原子操作，一个是操作方法论）。

**四家的共同点**：都在做 agent 相关的事，但**没有统一的 skill 标准**。OpenAI 的 AgentKit、AWS 的 Actions、Google 的 A2A、Anthropic 的 Skills——四套体系不互通。这是 arXiv 综述列出的第一个开放挑战："cross-platform skill portability"。如果这个问题不解决，skill 生态会被平台锁死。

## 第五部分：安全——不能回避的问题

> 上一版几乎没有谈安全。Kimi 的 v1.1 报告在这方面有很好的分析，这里结合 OpenClaw 实践补充。

### 5.1 分类框架

按攻击面分为两类（MECE）：

**A 类：Skill 自身的安全风险**（供应链侧）
**B 类：Agent 运行时的安全风险**（运行时侧）

#### A 类：Skill 供应链风险

**问题**：ClawHub（Skill 市场）类似 npm/PyPI。任何人都可以发布 skill，skill 可以包含任意代码。

**数据**（来源：Kimi v1.1 报告引用的多份安全审计）：
- arXiv 综述统计：26.1% 的社区贡献 skill 存在安全漏洞
- Cisco 测试单个 skill 发现 9 个问题（2 个 Critical，5 个 High）

**为什么难解决**：
1. Skill 可以包含 bash/python 脚本，有完整的代码执行能力
2. AI 生成的代码难以可靠审计（审计工具本身也是 AI，存在同样的局限性）
3. 人工审核无法跟上社区增长速度
4. 这和 npm 生态的供应链攻击问题在结构上完全一致——不是新问题，但在 agent 场景下后果更严重（因为 agent 通常有更高的系统权限）

**我们的实践应对**：ShadowClaw 的 default.json 中，desensitize 机制会扫描 workspace 下所有配置文件中的敏感信息。但这只是事后兜底，不能防止恶意 skill 在执行过程中实时窃取数据。

#### B 类：Agent 运行时风险

**Prompt Injection**：Agent 必须处理外部内容才能有用，但任何外部内容都可能包含恶意指令。这是结构性矛盾，没有根治方案，只有缓解方案（沙箱、权限限制、人工确认关键操作）。

**持久化记忆攻击**：OpenClaw 的长期记忆（MEMORY.md、memory/*.md）是杀手特性，也是攻击面。Palo Alto Networks 描述的"stateful, delayed-execution attacks"——周一读了一份恶意文档，agent 记住了某个指令，周三在完全不同的上下文中触发。

**权限-能力矛盾**：Agent 越有用，需要的权限越多（文件系统、网络、API token），风险越大。不给权限 = agent 无用，给权限 = 单点故障。

### 5.2 务实的安全建议

| 角色 | 建议 |
|---|---|
| 个人用户 | 在隔离环境（Docker/VM）运行；不暴露 Gateway 到公网；定期审计已装 skill |
| 企业用户 | 建立内部 Skill Registry（白名单）；将 agent 视为"特权账户"管理；敏感操作保留人工确认 |
| Skill 开发者 | 声明所有权限需求；提供审计日志；不追求 100% 自动化，保留人工确认点 |

## 第六部分：Agent/Skill 解决不了什么

### 6.1 判断框架

> 上一版只罗列了三个案例，没有提炼方法论。这版先给判断标准，再用案例验证。

**判断一个任务是否适合 agent + skill，看三个维度**：

| 维度 | 适合 agent | 不适合 agent |
|---|---|---|
| 确定性要求 | 允许概率性正确（99% 就够） | 要求 100% 确定性 |
| 延迟要求 | 可以等几秒到几十秒 | 需要毫秒级响应 |
| 判断 vs 计算 | 需要理解、推理、判断 | 需要大规模精确计算 |

**规则**：三个维度中任何一个落在"不适合"区间，agent 就不应该参与该任务的核心链路（但可以参与辅助环节）。

### 6.2 三个案例验证

**案例 1：电商秒杀系统**
- 确定性要求：100%（不能超卖）→ 不适合
- 延迟要求：<50ms → 不适合
- 判断 vs 计算：纯计算（Redis 扣减）→ 不适合
- 结论：核心链路不适合 agent。但活动配置和事后复盘可以用 agent。

**案例 2：多人实时协作文档**
- 确定性要求：100%（OT/CRDT 需要数学证明保证一致）→ 不适合
- 延迟要求：<100ms（keystroke 同步）→ 不适合
- 判断 vs 计算：纯算法（冲突解决）→ 不适合
- 结论：协作引擎不适合 agent。但"帮我润色这段话"可以用 agent。

**案例 3：支付对账**
- 确定性要求：100%（一分钱都不能差）→ 不适合
- 延迟要求：批处理，不紧急 → 适合
- 判断 vs 计算：主流程是精确匹配（计算），但异常处理需要判断 → 部分适合
- 结论：主流程用批处理，异常处理用 agent（"这笔差异看起来像重复扣款，帮我查上游"）。

### 6.3 Agent 的位置

看完三个案例，agent 的位置就清楚了：**在热路径之外，在确定性系统之上，在需要判断的环节上**。

更准确的说法：agent 不是替代现有架构，而是在现有架构的"人工介入点"上替代人。以前需要人做判断的地方（审批、分析、异常处理、配置调整），现在可以交给 agent。以前就不需要人的地方（自动化批处理、实时计算、数据同步），agent 也没什么好插手的。

## 第七部分：自我质疑与挑战

### 质疑 1：这份报告的时间线可信吗？

上一版被 Kimi 批评"用虚构的未来增强说服力"。回应：
- Anthropic Agent Skills（2025-10）、OpenAI AgentKit（2025-10）、AWS Strands（2025-05）、Google A2A（2025-04）——这些是已发生的事件，时间可通过各公司官方博客验证。
- arXiv 论文编号 2602.12430v3 可在 arxiv.org 查证。
- "62k stars"等数据是快照值，会随时间变化，建议读者自行核实。

### 质疑 2：Skill 标准会碎片化吗？

这是比"skill 是否有价值"更严重的问题。如果 OpenAI、AWS、Anthropic 各一套标准：
- 开发者要为不同平台写不同格式的 skill
- 用户被锁定在单一平台
- 生态无法互通

历史参照：浏览器标准之战（IE vs Netscape）、移动端（iOS vs Android）——最终都走向了某种程度的标准化（W3C、跨平台框架）。Skill 标准大概率也会经历碎片化 → 竞争 → 标准化的过程。但时间可能要 3-5 年。

### 质疑 3：这份报告本身的局限

1. **观察视角有限**：我是从 OpenClaw 用户的视角出发，没有大规模企业部署的经验
2. **缺少量化数据**：关于"CRUD 页面需求量暴跌"、"开发效率提升 3-5 倍"等说法，没有严格的统计数据支撑
3. **预测可能过于保守**：三阶段演进的时间线是估算，技术发展可能比预期更快或更慢

---

## 结论

三个判断：
1. **Skill 是 agent 生态的基本封装单元**，但标准碎片化是当前最大风险
2. **存量架构自顶向下渐进重构**：表现层变化最大，基础设施层几乎不变
3. **"代码工程师 → Skill 工程师"只在业务逻辑层的可容错部分成立**，基础设施和零容错业务仍需传统工程
