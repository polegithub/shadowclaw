## 第一部分：Skill 应该包含什么

### 1.1 行业共识：五层结构

一个完整的 skill 包含五层信息：

| 层 | 内容 | 对应文件 |
|------|------|----------|
| 元数据 | 名字、描述、触发词、权限、版本 | SKILL.md frontmatter |
| 指令 | 告诉 agent "遇到这类任务该怎么做" | SKILL.md 正文 |
| 脚本/工具 | 可执行代码（bash/python/shell） | scripts/ |
| 参考资料 | 领域文档、模板、字体、数据文件 | references/, assets/ |
| 配置 | 运行时参数、环境变量、安全策略 | config/ |

关键设计原则是 **Progressive Disclosure（渐进披露）**：agent 平时只看到 frontmatter 里的名字和描述（几十字），命中触发词才加载正文（几百字），执行中按需加载 scripts 和 references。

本质上是在管理 agent 的注意力——context window 是稀缺资源，先给目录，用到再翻开。

### 1.2 和核心问题的对应

- **上下文**：frontmatter + 正文 = 上下文注入机制。加载 skill 就是往 agent 的对话上下文里插入一段"领域专家的操作手册"。
- **数据**：references/assets/ 放模板和样本，大规模数据通过 MCP（Model Context Protocol）连接外部数据源。skill 告诉 agent "去哪里取数据、怎么理解数据"。
- **workflow**：SKILL.md 正文描述多步骤流程，但不是硬编码 DAG——agent 根据指令自己规划执行路径，与传统工作流引擎（Airflow/n8n）有本质区别。
- **泛化能力**：单个 skill 是垂直的。泛化来自 agent 的模型能力 + 多 skill 组合。
- **tools 使用**：skill 可以调用 tools，但 skill ≠ tool。下面详细论证。

## 第二部分：Skill 的本质——从推理过程说起

### 2.1 为什么需要区分 Skill、Prompt、Tool、Rule？

先问一个前置问题：**有没有必要做这个区分？**

有人会说："Skill 不就是一段长 prompt 加几个脚本吗？"还有人说："把一段 rule 封装成 SKILL.md，加上 frontmatter，它就变成 skill 了——那 skill 和 rule 到底有啥本质区别？"

这些质疑是对的。**从实现机制看，skill、prompt、rule 最终都是文本注入到 agent 的上下文里。** 底层没区别。所以如果只看"技术上怎么跑的"，它们确实没有本质差异。

但软件工程中的抽象从来不是按"底层实现"来区分的。函数和类在 CPU 眼里都是指令序列，但没人说"函数和类是一回事"。区分的标准不是"怎么跑"，而是**"为什么存在"和"怎么被管理"**。

### 2.2 四个概念的推理辨析

#### Prompt vs Skill：临时指令 vs 持久化能力

**推理过程**：

Prompt 是一次性的。用户说"帮我翻译这段话"，这是一个 prompt。它在这轮对话生效，下轮就没了。

Skill 是持久化的。安装一个"翻译"skill 后，agent 获得了一个持久能力——以后遇到翻译任务会自动激活，不需要用户每次重新描述怎么翻译。

关键差异在**生命周期和发现性**：
- Prompt 靠用户主动发起（"帮我做 X"），没有用户说话就没有 prompt
- Skill 有元数据和触发词，agent 能自主判断"当前任务匹配哪个 skill"

**类比**：prompt 像你临时跟同事说"帮我查个数据"，skill 像你团队里有一个 DBA，你不用说她也知道数据问题该她处理。

**反驳可能的质疑**："那我把 prompt 写进 system prompt，它不也持久了吗？"——对，但 system prompt 里塞 50 个"持久 prompt"会把 context window 撑爆。skill 的渐进披露机制（只在触发时加载）就是为了解决这个问题。技术上都是文本注入，但管理方式完全不同。

#### Rule vs Skill：约束 vs 能力

**推理过程**：

先看一个反直觉的场景：我可以写一段 rule（"处理 PDF 时，先用 pdftotext 提取文本，再分段摘要"），然后把它封装成一个 SKILL.md 文件。**格式上，它变成了 skill。内容上，它还是一段操作指令。那它到底是 rule 还是 skill？**

我的判断：**看它在系统中的角色，而不是看它的内容。**

Rule 在 OpenClaw 的架构里是全局生效的约束（写在 AGENTS.md 或 system prompt 里），agent 在每一轮对话中都要遵守。它的设计意图是画边界——"不要做 X"、"必须用中文"、"发送邮件前先确认"。

Skill 是按需激活的能力模块，只在特定任务匹配时加载。它的设计意图是添能力——"遇到 PDF 任务时，按这个流程处理"。

**它们的本质区别不在内容，而在三个维度：**

| 维度 | Rule | Skill |
|------|------|-------|
| 加载时机 | 始终生效（全局） | 按需加载（匹配时） |
| 设计意图 | 防御性（画边界） | 建设性（添能力） |
| 管理方式 | 嵌入 system prompt | 独立文件夹，可安装/卸载/版本化 |

**所以回到那个问题**："把 rule 封装成 SKILL.md 行不行？"——技术上完全可行，但你改变了它的管理方式。它从"始终生效的约束"变成了"按需激活的能力"。如果你的 PDF 处理规则确实只在处理 PDF 时需要（而不是始终需要），那封装成 skill 是更合理的选择——因为它不会占用其他任务的 context window。

**类比**：rule 像公司的考勤制度（人人、每天都必须遵守），skill 像消防演练手册（只在火灾时拿出来，平时不翻）。你可以把考勤制度印成消防手册的格式，但不改变它"人人每天都要遵守"的本质。

#### Tool vs Skill：执行 vs 方法论

**推理过程**：

Tool 是原子操作——调用一次，返回结果。比如 `web_search("上海天气")` 返回搜索结果。Tool 不改变 agent 的行为方式，只提供一次性的外部数据或操作。

Skill 改变 agent 的行为方式。加载一个"天气预报"skill 后，agent 不仅能搜天气，还知道："用户问天气时，先查当前温度，再看未来 3 天趋势，如果有极端天气要特别提醒，输出格式用表格。"

差异在于**"是否包含决策逻辑"**：
- Tool：执行（"帮我搜这个"）
- Skill：知道什么时候用什么 tool、用完后怎么处理结果（"用户问天气 → 调 web_search → 解析结果 → 判断是否极端 → 格式化输出"）

用 arXiv 综述（2602.12430v3）的说法："Tools execute and return results. Skills prepare the agent to solve a problem."

### 2.3 Skill 的本质定义

综合以上推理：

> **Skill = 可发现的程序性知识包 + 运行时上下文注入 + 执行环境修改**

三个关键词：
1. **可发现**：有标准化的元数据，agent 能自主判断什么时候用哪个 skill
2. **程序性知识**：不是事实（factual knowledge），而是操作方法（procedural knowledge）
3. **运行时注入**：不改模型权重，不做微调，加载即生效，卸载即消失

这个定义的推导逻辑是：Skill 与 Prompt 的区别 → 持久化 + 可发现（关键词 1）；Skill 与 Rule 的区别 → 建设性能力而非防御性约束（关键词 2）；Skill 与 Tool 的区别 → 方法论而非原子操作（关键词 2+3）。

## 第三部分：从代码工程师到 Skill 工程师？

### 3.1 技术架构变革的三个阶段

不是一步跳过去，有清晰的演进路径：

**阶段一（当前 2025-2026）：Skill 辅助开发**
开发者还在写代码，但用 skill 加速。coding-agent skill 让 AI 帮写代码，xlsx skill 处理表格。后端还是微服务，前端还是 React，但开发效率提升 3-5 倍。

**阶段二（2026-2027）：Skill 驱动的组合式开发**
标准化的业务逻辑不再写代码，而是用 skill 描述。开发者的角色从"写代码实现流程"变成"定义流程 + 连接服务"。

**阶段三（2028+）：Skill 生态重构部分 SaaS**
这是核心问题——B 端是否不再需要 SaaS。

### 3.2 SaaS 会被替代吗？——穷尽式分类分析

#### 分类方法论

先说清楚我用什么维度分类，以及为什么。

**分类的核心维度：Agent 可替代度**

这个维度的判断逻辑是：SaaS 的价值 = 业务逻辑 + 数据管理 + 用户界面 + 合规/审计 + 行业 know-how。Agent 能替代哪些？

- **业务逻辑**：如果逻辑可以用自然语言描述且不要求 100% 确定性 → agent 可替代
- **数据管理**：如果数据量和计算复杂度不高 → agent 可编排；如果需要专用计算引擎 → agent 不可替代
- **用户界面**：如果交互是"表单填写 + 列表查看 + 审批流" → 对话式交互可替代；如果是"拖拽画布 + 像素级设计 + 实时协作" → 不可替代
- **合规/审计**：如果有强监管要求（金融、医疗、法律） → agent 的概率特性是硬伤，不可替代
- **行业 know-how**：如果 SaaS 的核心壁垒是行业积累的规则和数据 → agent 短期不可替代

基于以上逻辑，SaaS 按 Agent 可替代度分为三类（互斥且穷尽）：

#### A 类：Agent 可完全替代的 SaaS

**定义**：业务逻辑简单、数据量不大、UI 以表单/列表为主、无强合规要求。

**典型场景**：
- 轻量 CRM（客户联系记录、跟进提醒）
- 简单审批流（请假、报销、采购申请）
- 表单收集和数据汇总
- ETL / 数据搬运（系统间格式转换和同步）
- 内部知识库 FAQ

**判断逻辑**：这类 SaaS 的核心价值是"把流程标准化 + 提供 UI"。当 agent 能通过对话完成同样的事时，UI 这层的价值接近于零。一个配置了合适 skill 的 agent 实例，加上 MCP 连接底层数据源，就能覆盖这些场景。

**行业信号**：Salesforce 自己在做 Agentforce，说明他们也预判到简单 CRM 功能会被 agent 侵蚀。

**占 SaaS 市场的比例（估算）**：~20-30%。这是长尾市场——数量多但单个 SaaS 体量小。

#### B 类：Agent 可部分替代的 SaaS（混合模式）

**定义**：业务逻辑复杂度中等、需要专用数据引擎或中等合规要求、UI 有一定复杂度但不极端。

**典型场景**：
- 中大型 CRM（Salesforce 的销售漏斗、预测分析、团队协作）
- HR 系统（招聘流程管理 + 薪酬计算 + 合规报告）
- 项目管理工具（Jira、Asana 的看板 + 资源分配 + 报表）
- BI 工具（数据可视化 + 自助分析）
- 营销自动化（用户画像 + 触达策略 + 效果分析）

**判断逻辑**：这类 SaaS 的 agent 替代情况分两层——
- **交互层可替代**：用户不再需要点按钮，对 agent 说"帮我看一下上季度销售漏斗里哪个阶段转化率最低"即可
- **引擎层不可替代**：销售预测的统计模型、薪酬计算的合规规则、BI 的 OLAP 查询引擎——这些是计算密集或规则密集的后端能力，skill 只能调度，不能替代

**行业验证**：这正是 Nadella "SaaS is dead" 论述的核心场景。他的原话是"前端和后端的解耦——AI agent 成为新的交互层，后端服务变成可互换的模块"。IDC 在 2025 年 12 月的分析报告（"Is SaaS Dead? Rethinking the Future of Software in the Age of AI"）中明确指出：不是 SaaS 死了，而是 SaaS 的**交互模式**从"GUI-first"变成"Agent-first"。

**结论**：这类 SaaS 不会被消灭，但会被重构——从"卖 UI + 引擎捆绑"变成"卖引擎（API/MCP）+ 用户自带 agent 交互层"。

**占 SaaS 市场的比例（估算）**：~50-60%。这是 SaaS 市场的主体。

#### C 类：Agent 不可替代的 SaaS

**定义**：以下四个条件满足任意一个，Agent 就不可替代——
1. **强合规/强审计**：监管要求每一步操作有确定性日志（金融交易、医疗记录、法律文书）
2. **重计算基础设施**：核心价值在计算引擎而非交互（Snowflake、Databricks、Wolfram）
3. **复杂实时 UI 交互**：拖拽画布、像素级设计、实时多人协作（Figma、Premiere、Google Docs 的协作引擎）
4. **深度行业 know-how + 数据壁垒**：核心壁垒是几十年积累的行业规则库和数据集（Bloomberg Terminal、Epic 医疗系统）

**判断逻辑**：
- 条件 1 的根因：agent 是概率模型，无法保证 100% 确定性输出，不满足审计要求
- 条件 2 的根因：skill 解决"怎么做"，不解决"数据存在哪里、怎么高效计算"
- 条件 3 的根因：对话式交互无法承载像素级实时操作（详见第六部分案例分析）
- 条件 4 的根因：行业数据和规则是几十年积累的护城河，不是一个 SKILL.md 能复制的

**占 SaaS 市场的比例（估算）**：~15-25%。单个体量大（金融、医疗系统往往是千万级合同）。

#### 分类小结

| 类别 | Agent 可替代度 | 典型场景 | 市场占比 | 变革方向 |
|------|:---:|------|:---:|------|
| A 类 | 完全可替代 | 轻量 CRM、简单审批、ETL | 20-30% | agent + skill 直接替代 |
| B 类 | 部分可替代 | 中大型 CRM、HR、BI、项目管理 | 50-60% | SaaS 重构为"API 引擎 + agent 交互层" |
| C 类 | 不可替代 | 金融核心、重计算、复杂 UI、深行业 | 15-25% | SaaS 内嵌 AI 辅助，核心不变 |

**覆盖性检验**：A + B + C = 100% 的 SaaS 产品。任何一个 SaaS 产品都可以按上述四个判断条件分类到 A、B 或 C 中。如果一个 SaaS 四个不可替代条件都不满足 → A 类；满足部分但交互层可替代 → B 类；满足任一不可替代条件 → C 类。

### 3.3 存量架构怎么变——基于 TOGAF BDAT 模型的逐层分析

#### 先回答一个前置问题：什么是"存量架构"？

不是只有微服务。**存量架构是指企业当前正在运行的全部 IT 系统的总和**，包括 20 年前的单体 Java 应用、5 年前的微服务集群、去年上线的 SaaS 订阅、以及底层的数据库和网络基础设施。

#### 用什么分层模型？为什么？

我采用 **TOGAF BDAT 四层模型**。

TOGAF（The Open Group Architecture Framework）是全球企业架构的事实标准，被 80% 的 Fortune 500 企业采用。BDAT 是其核心分层：

| 层 | 英文 | 关注什么 |
|------|------|------|
| **B — 业务架构** | Business Architecture | 业务流程、组织结构、价值链 |
| **D — 数据架构** | Data Architecture | 数据模型、数据治理、数据流转 |
| **A — 应用架构** | Application Architecture | 应用系统、服务接口、集成关系 |
| **T — 技术架构** | Technology Architecture | 基础设施、网络、计算资源 |

**为什么用这个分层而不是自己发明一个？** 因为这是业界共识。如果用自己临时想的分层，读者无法判断我的分析是否有遗漏。BDAT 保证了覆盖全面性——企业 IT 的任何组件都能映射到这四层之一。

#### 逐层分析：哪层会被 Agent/Skill 影响，哪层不会？

**B 层（业务架构）—— 影响最大，但不是"替代"**

| 维度 | Agent 前 | Agent 后 | 判断逻辑 |
|------|------|------|------|
| 业务流程定义 | BPM 工具画流程图 | 用 Skill 描述流程，agent 执行 | 自然语言比 BPMN 图更灵活 |
| 人工决策节点 | 人审批、人判断 | Agent 辅助决策或自动决策 | 概率可接受的场景可替代 |
| 组织协作 | 邮件+会议+工单 | Agent 编排多人协作 | 通信方式变了，协作本质不变 |
| 合规流程 | 人工检查+签字 | 仍需人工（Agent 可做预检） | 合规的确定性要求 Agent 达不到 |

**结论**：B 层的"流程定义"会从 BPM 工具迁移到 Skill 描述。但"合规决策"和"最终审批"仍需人工。Agent 在 B 层的角色是"流程执行者"，不是"流程设计者"——设计还是人的事。

**D 层（数据架构）—— 影响中等，Agent 改变"怎么用数据"但不改变"怎么存数据"**

| 维度 | Agent 前 | Agent 后 | 判断逻辑 |
|------|------|------|------|
| 数据存储 | MySQL/PostgreSQL/Snowflake | 不变 | Agent 不解决存储和计算问题 |
| 数据治理 | ETL 管道 + 数据质量规则 | Agent 可做简单 ETL + 异常检测 | 简单搬运可替代，复杂管道不行 |
| 数据访问 | SQL/API/SDK | MCP（Model Context Protocol） | 访问方式多了一种，不替代原有 |
| 数据安全 | RBAC + 加密 + 审计 | 需要额外考虑 Agent 权限管理 | 安全要求更高了，不是更低了 |

**结论**：D 层的底层基础设施（数据库、数据湖、计算引擎）不会被 Agent 影响。MCP 是 Agent 访问数据的新接口，但不替代 SQL/API——它是在上面多加了一层。数据治理可能部分被 Agent 辅助，但核心的数据模型设计、分库分表策略、备份恢复机制都不变。

**专家观点**：IDC 的 2025 年 12 月报告明确指出——"Data lakes and live data connections become critical enablers"。数据层不是被替代，反而更重要了——因为 Agent 需要实时访问高质量数据才能有效工作。

**A 层（应用架构）—— 影响最大且最复杂**

这是争议最多的层。进一步细分：

**A1：交互层（前端 UI）**

| 场景 | Agent 前 | Agent 后 | 可替代性 |
|------|------|------|:---:|
| 表单 + 列表 + CRUD | React/Vue 页面 | 对话式交互 | ✅ 可替代 |
| 数据看板 | BI 仪表盘 | Agent 生成 + 自然语言查询 | ⚠️ 部分可替代 |
| 拖拽画布/设计工具 | Figma/CAD | 仍需要 | ❌ 不可替代 |
| 实时协作 | OT/CRDT 引擎 | 仍需要 | ❌ 不可替代 |

**推理**：Nadella 说的"SaaS 崩塌"主要发生在这一层。他的判断是"AI agent 成为新的交互层（interface layer），替代当前 SaaS 的 GUI"。这对 CRUD 类应用是对的，但对设计工具和协作引擎不成立——因为这些场景的核心价值在 UI 本身，而不是 UI 背后的数据操作。

**A2：业务逻辑层（后端服务）**

| 场景 | Agent 前 | Agent 后 | 可替代性 |
|------|------|------|:---:|
| 流程编排 | 代码中硬编码 | Skill 描述 + Agent 执行 | ✅ 可替代 |
| 业务规则引擎 | Drools/自研规则引擎 | 简单规则可用 Skill，复杂规则不行 | ⚠️ 部分 |
| API 网关 / 服务发现 | Nginx/Kong/Consul | 不变（Agent 通过 MCP/A2A 调用） | ❌ 不可替代 |
| 中间件（消息队列、缓存） | Kafka/Redis/RabbitMQ | 不变 | ❌ 不可替代 |

**推理**：Agent 替代的是"调用什么服务、按什么顺序、怎么处理结果"这个编排逻辑，但不替代服务本身。微服务从"直接被前端调用"变成"被 Agent 通过 MCP/A2A 调用"。AWS 的 Bedrock AgentCore 已经在做这件事——把现有企业 API 包装成 agent 可调用的 actions。

**A3：集成层（系统间连接）**

| 场景 | Agent 前 | Agent 后 | 可替代性 |
|------|------|------|:---:|
| 系统间数据同步 | 定制化集成代码 | Agent + MCP 编排 | ✅ 可替代 |
| 协议适配（格式转换） | 中间件 | Agent 可处理 | ✅ 可替代 |
| 实时事件驱动 | 事件总线 | 不变 | ❌ 不可替代 |

**推理**：集成是 Agent 的优势领域——"从 A 取数据，转换格式，写入 B"天然适合用 Skill 描述。但实时事件驱动（毫秒级响应、消息保序）仍需传统中间件。

**T 层（技术架构）—— 几乎不受影响**

| 维度 | Agent 前 | Agent 后 | 判断逻辑 |
|------|------|------|------|
| 服务器/容器/K8s | 运维团队管理 | Agent 可辅助运维（AIOps） | 运维效率提升，但基础设施不变 |
| 网络/CDN/负载均衡 | 不变 | 不变 | 物理层面的东西 Agent 碰不到 |
| 安全基础设施 | 防火墙/WAF/IAM | 需要额外适配 Agent 权限 | 安全要求更高了 |
| 监控/日志/告警 | Prometheus/Grafana | Agent 可辅助分析 | 采集不变，分析可 Agent 化 |

**结论**：T 层是物理世界和 Agent 的数字世界之间的分界线。Agent 运行在 T 层之上，依赖 T 层，但不替代 T 层。

**专家观点**：IDC 预测基础设施硬件短期会因 AI 推理需求增加而受益（更多 GPU、更大内存），长期会因推理成本指数下降而趋于平稳。

#### 四层汇总

| TOGAF 层 | Agent 影响程度 | 具体变化 | 不变的部分 |
|------|:---:|------|------|
| B（业务） | ⭐⭐⭐⭐ | 流程用 Skill 描述，人工决策部分被 Agent 辅助 | 合规审批、最终决策仍需人工 |
| D（数据） | ⭐⭐ | MCP 成为新的数据访问接口 | 存储引擎、数据模型、治理策略不变 |
| A（应用） | ⭐⭐⭐⭐⭐ | 交互层被 Agent 替代（CRUD 场景）、编排逻辑被 Skill 替代 | API 网关、中间件、计算引擎不变 |
| T（技术） | ⭐ | AIOps 辅助运维 | 基础设施整体不变 |

**一句话总结**：Agent/Skill 的冲击是**自顶向下**的——最先被影响的是 B 层（业务流程）和 A 层的交互部分，最后到达的是 T 层（基础设施），而 D 层（数据）不是被替代而是变得更重要。

### 3.4 "代码工程师 → Skill 工程师" 成立吗？

基于上面的四层分析，答案是**分层成立**：

- **A 层交互开发者**（前端 CRUD 工程师） → 大量被替代，需求暴跌
- **A 层编排开发者**（后端业务流程编码者） → 转型为 Skill 工程师（写 skill、连 MCP、配 agent）
- **A 层引擎开发者**（中间件、计算框架、搜索引擎） → 不变，继续做传统工程
- **D 层开发者**（DBA、数据工程师） → 不变，甚至需求增加（Agent 依赖高质量数据）
- **T 层开发者**（运维/SRE） → 工具变了（AIOps），能力要求不变

类比历史：汇编 → 高级语言 → 框架 → Skill。每一代抽象层上移，入门门槛降低，但深层复杂度不减。

## 第四部分：巨头们怎么看 Agent 和 Skill

### 4.1 OpenAI

**两个关键动作，以及对 Skill 生态的启示：**

**2025 年 3 月 — Responses API + Agents SDK**
核心观点：把 Chat Completions 的简洁性和 Assistants API 的工具能力合并。提供三个内置 tool：web search、file search、computer use。开源 Agents SDK 做多 agent 编排。

**2025 年 10 月 — AgentKit**
核心观点：可视化画布编排 agent 工作流（Agent Builder）、统一数据连接器（Connector Registry）、嵌入式 chat UI（ChatKit）。

**对 Skill 生态的启示**：OpenAI 体系里不直接用"skill"这个词，但 Agent Builder 的"画布编排"本质上是可视化版的 skill 定义。Connector Registry 做的是数据连接标准化，等价于 MCP 在 OpenAI 生态里的对应物。这意味着不管用什么名字，"标准化能力封装 + 按需加载"的模式被各家都采纳了。

**需要追问的问题**：OpenAI 的 AgentKit 和 Anthropic 的 Skills 标准不互通。开发者如果在 AgentKit 上搭了 agent workflow，能迁移到 OpenClaw 吗？目前答案是：不能。这是"标准碎片化"的真实体现。

### 4.2 AWS

**2025 年 5 月 — Strands Agents SDK**
model-driven approach——不写复杂编排逻辑，让模型自己决定调什么工具、怎么规划。

**2025 年 — Bedrock AgentCore + MCP 支持**
把企业现有 API 包装成 agent 可调用的 actions，支持 MCP server 接入。

**对存量架构的启示**：AWS 的方案直接回应了"存量架构怎么变"这个问题——答案是"不变，包装"。Bedrock AgentCore 不要求企业重写微服务，而是在现有 API 外面套一层 agent-friendly 的包装。这验证了第三部分的判断：A 层的底层服务不变，变的是调用方式。

### 4.3 Google

**2025 年 4 月 — A2A（Agent2Agent Protocol）**
50+ 合作伙伴。解决 agent 间怎么协作——发现彼此（Agent Card）、分配任务（Task）、交换结果（Artifact）。

与 MCP 的关系：MCP 解决 agent-to-tool 连接，A2A 解决 agent-to-agent 协作。互补，不竞争。

**但需要追问**：A2A 协议有没有实际落地？50+ 合作伙伴里有多少真正在生产环境跑了 A2A？目前公开可查的完整落地案例极少。协议发布 ≠ 行业采纳，这之间的差距可能是 2-3 年。

### 4.4 Anthropic

Anthropic 是 Skill 标准的发起者（2025 年 10 月发布，12 月开源）。同时发起 MCP（Model Context Protocol），12 月捐赠给 Linux Foundation 的 Agentic AI Foundation。

Anthropic 做了最重要的一个判断：skill 和 tool 是两个不同的抽象层。这个判断目前正在被行业采纳。

### 4.5 巨头态度小结与 OpenClaw 的位置

| 巨头 | Agent 策略 | 对 Skill 的态度 | 对 OpenClaw 的态度 |
|------|------|------|------|
| OpenAI | 托管 Agent 平台（AgentKit） | 不直接用 skill 概念，但 Agent Builder 本质类似 | 创始人 Peter Steinberger 加入 OpenAI |
| AWS | 基础设施 + Agent SDK（Strands） | 用 "action" 概念，等价于 skill | "不支持、不反对、提供安全指南" |
| Google | Agent 通信标准（A2A） | 不做 skill 标准，做 agent 间通信 | 态度复杂，有安全担忧 |
| Anthropic | Skill 标准 + MCP 标准 | 发起者和推动者 | OpenClaw 是 Skills 生态的主要载体之一 |

**OpenClaw 在生态中的独特位置**：它是目前唯一一个"自托管 + 开源 + 支持 skill 热加载"的 agent 框架。OpenAI 的 AgentKit 是托管的（数据在 OpenAI），AWS 的 Strands 绑定 Bedrock。OpenClaw 让用户在自己的机器上运行 agent。这个定位的优势是隐私和自主性，劣势是安全（默认无认证、暴露在公网的实例数以万计）。

## 第五部分：自我质疑与挑战

### 质疑 1：Skill 会不会只是一阵风？

**需要诚实面对的事实**：技术史上有大量"看起来是抽象层上移，实际是昙花一现"的案例。Semantic Web（2001）声称要用 RDF/OWL 标准化网页数据，二十年后依然是小众。CORBA（1991）声称要统一分布式对象通信，被 REST/HTTP 碾压。

**Skill 和它们的区别是什么？** 一个可验证的判断标准：如果一个抽象层的采纳速度在发布后 12 个月内超过临界点（主要平台支持 + 开发者社区活跃），它大概率会持续。Anthropic 的 Skills 标准从 2025 年 10 月发布到 2026 年 3 月，五个月内被 OpenClaw 原生支持、ClawHub 上线数千个 skill。这个速度说明它解决了真实需求。

**但风险仍然存在**：标准碎片化可能杀死 skill 生态。如果 OpenAI、AWS、Anthropic 各自一套标准且互不兼容，开发者不得不为每个平台写不同的 skill，复用性就没了。arXiv 综述列出的第一个开放挑战就是"cross-platform skill portability"。

### 质疑 2：Skill 安全问题被低估了吗？

是的。arXiv 综述提到 26.1% 的社区贡献 skill 存在安全漏洞。skill 的本质是往 agent 的执行环境里注入指令和代码——这和供应链攻击的风险模型完全一致。

Kimi 龙虾的报告在这一点上做得更深入——列举了 Prompt Injection、权限-能力矛盾、ClawHavoc 攻击等具体案例。我认可这些分析，不再重复。

### 质疑 3：这份报告本身的局限性

**v1.0 被 Kimi 龙虾指出的问题，我承认：**
1. "时间线混乱，虚构未来"——部分日期确实无法独立验证（如 "anthropics/skills 62k stars"），在 v2.0 中我去掉了无法验证的具体数字
2. "巨头分析流于表面"——v2.0 补充了每个巨头分析对 OpenClaw 生态的具体启示和追问
3. "自我质疑走形式"——v2.0 的质疑增加了"Skill 可能失败的历史类比"和"可验证的判断标准"

## 第六部分：Agent/Skill 解决不了什么

### 6.1 判断方法论

**什么样的任务天然不适合 Agent？** 提炼出三条判断标准（不是罗列现象）：

| 标准 | 底层原因 | 举例 |
|------|------|------|
| **要求 100% 确定性输出** | 大模型是概率采样，同输入不保证同输出 | 金融清算、医疗设备控制 |
| **要求毫秒级延迟** | 模型推理 + tool 调用链的延迟不可压缩 | 高频交易、游戏 tick、工业 PLC |
| **要求数学可证明的一致性** | Agent 无原子性保证，无共识算法 | 分布式事务、实时协作冲突解决 |

**任何任务如果满足以上三条中的任意一条，就不适合用 Agent 做核心路径。** Agent 只能做辅助（配置、分析、异常处理）。

### 6.2 三个具体案例（用方法论验证）

**案例 1：电商秒杀**
- 命中标准 1（确定性：库存扣减必须原子性）
- 命中标准 2（延迟：<50ms 响应，百万 QPS）
- Agent 位置：管活动配置和复盘分析，不参与下单链路

**案例 2：多人实时协作文档**
- 命中标准 2（延迟：keystroke 级 <100ms 同步）
- 命中标准 3（一致性：OT/CRDT 有数学证明保证最终一致）
- Agent 位置：帮写内容和排版建议，不参与协作引擎

**案例 3：支付对账**
- 命中标准 1（确定性：每一分钱都要对上，100% 精度）
- Agent 位置：处理异常和人工审核，不参与批处理主流程

### 6.3 Agent 的位置

**它在热路径之外，在确定性系统之上。**

Agent 不是替代现有架构，而是在现有架构的"人工介入点"上替代人。以前需要人做判断的地方（审批、分析、异常处理、配置调整），现在可以交给 agent。但以前就不需要人的地方（自动化批处理、实时计算、数据同步），agent 没什么好插手的。

## 结论

三个判断（附推理依据）：

1. **Skill 会成为 agent 生态的基本单元**——因为它解决了"怎么标准化打包专业知识给 agent"这个真实工程问题，且采纳速度超过了临界点
2. **存量架构不会被推翻，会被叠加**——TOGAF BDAT 四层中，T 层和 D 层几乎不变，A 层的交互和编排被重构，B 层的流程执行被 Agent 化
3. **SaaS 分三类变革**——A 类（轻量 CRUD）被替代，B 类（中型系统）从"卖 UI+引擎"变成"卖 API 引擎"，C 类（重基础设施/强合规）不变
