## 一、背景

将整个后端体系（API、数据库、运维、公共服务、基建）从传统架构改造为 AI 友好的状态，使 AI Agent 成为系统的第一类用户，能理解业务、调用服务、独立完成研发闭环。

---

## 二、核心判断

### 不是重写，是"长出接口"

在现有架构上长出一层 AI 可操作的接口层，让 AI Agent 能像一个新入职的后端工程师一样，通过标准化方式理解系统、调用服务、排查问题。

### Skills vs MCP — 两个都要，定位不同

|  | MCP Server | Skills |
|---|---|---|
| **本质** | 标准化工具接口协议 | Agent 的可执行知识包 |
| **类比** | USB 接口 | 驱动 + 说明书 |
| **内容** | 暴露原子能力：查、改、调 | 包含上下文：什么时候用、怎么组合、有什么坑 |
| **谁调** | 任何支持 MCP 的 Agent | 特定 Agent 框架（如 OpenClaw） |
| **适合** | 底层能力暴露 | 业务场景编排 |

**底层用 MCP 暴露能力，上层用 Skills 编排场景。**

MCP 只解决"AI 能调到"，不解决"AI 知道什么时候该调、怎么组合调"。Skills 补的是这个。

## 三、架构分层

```
┌─────────────────────────────────────────────┐
│  Skills 层（场景编排）                        │
│  跨仓库、跨服务的业务流程串联                   │
├─────────────────────────────────────────────┤
│  业务知识层（AI 理解业务的关键）                │
│  行业配置 / 实验规则 / 上下游依赖 / 业务 SOP    │
├─────────────────────────────────────────────┤
│  业务 API 层（3 个仓库）                      │
│  各自独立，各自暴露 MCP Server                 │
├─────────────────────────────────────────────┤
│  公共服务 API 层                              │
│  用户中心 / 支付 / 消息 等，也暴露 MCP          │
├─────────────────────────────────────────────┤
│  基建层（运维 / 网络 / DB）                    │
│  K8s 操作 / DB 查询 / 日志 / 监控 → MCP Server │
└─────────────────────────────────────────────┘
```

### 每一层做的事

**基建层 → MCP Server**
- DB 查询（只读）、日志检索、监控指标、部署/回滚操作
- 纯工具，无业务语义

**公共服务层 → MCP Server**
- 用户查询、权限校验、消息推送等原子接口
- 每个公共服务一个 MCP Server

**业务 API 层（3 个仓库）→ MCP Server + 轻量 Skills**
- 每个仓库暴露自己的 MCP Server
- 同时写一组 Skills 描述核心业务场景

**业务知识层 → 结构化知识库（AI 可查询）**
- 行业配置矩阵（哪些行业支持哪些功能、有什么特殊规则）
- 实验冲突规则（优先级、互斥关系）
- 上下游依赖关系（接口字段、前置条件）
- 这不是文档，是 AI 通过 MCP 直接查询的结构化数据

**Skills 层 → 场景编排**
- 跨仓库、跨服务的业务流程
- 把人跨系统操作的 SOP 变成 AI 可执行的程序

## 四、改造阶段

### 阶段 A：让 AI 能"操作"系统
- 核心接口包装成 MCP Server / Tool Spec，AI Agent 可直接调用
- 运维操作（部署、回滚、扩缩容、数据修复）封装成原子化命令
- 加护栏：操作分级（只读/可写/危险），危险操作需人工审批

### 阶段 B：让 AI 能"改进"系统
- CI/CD 流水线对 AI 开放（AI 提 PR → 跑测试 → 人审批合入）
- 测试覆盖率拉上去，AI 改完代码能自验证
- 代码规范、架构约束文档化（AGENTS.md），AI 改代码自动遵循

### 阶段 C：业务知识结构化
- 把散落在产品脑子里、代码 if-else 里、聊天记录里的业务规则，结构化录入
- 行业配置、实验规则、上下游依赖全部可查询
- 这是最重要也最耗人力的阶段

## 五、终态

### 用户看到的
两种模式共存，共享同一套后端：
- **传统界面**：高频、确定性操作，点按钮更快（查列表、改状态）
- **AI 驱动**：非标准、跨系统、模糊意图的操作，说人话完成

前端界面不会消失，但会越来越薄。很多现在需要专门开发页面的功能，未来一个 Skill 就能代替。真正减少的是"为了操作数据而写的前端页面"，留下来的是"为了信息展示和用户体验必须存在的界面"。

### 角色变化

| | 现在 | 终态 |
|---|---|---|
| **产品** | 写 PRD → 等排期 → 等开发 → 验收 | 说需求 → AI 确认 → 几小时后验收 |
| **研发** | 接需求 → 写代码 → 联调 → 测试 → 部署 | 审方案 → 审代码 → 批上线 → 处理 AI 搞不定的 |
| **AI** | 不存在 | 需求拆解 → 编码 → 测试 → 部署 → 运维 |

研发从"写代码的人"变成"审代码的人 + 建基础设施的人"。产品的体感：需求从"周级"变成"天级"甚至"小时级"。

### 终态架构
```
人类（定义意图、审批、兜底）
       ↓
AI 编排层（理解意图 → 拆解任务 → 调度执行）
       ↓
Tool/MCP 层（标准化接口，所有能力原子化暴露）
       ↓
业务知识层（行业规则、实验配置、上下游依赖）
       ↓
服务层（微服务，每个服务自带 AI 可读文档）
       ↓
基础设施（DB/MQ/K8s，AI 可观测 + 可操作）
```

## 六、需求开发流程模拟（终态）

**需求：商家优惠券批量发放接口，5000 QPS，幂等，失败可重试。**

| 时间 | 阶段 | 谁做 | 做什么 |
|---|---|---|---|
| 09:30 | 需求输入 | 研发 | 输入需求描述 |
| 09:31 | 技术调研 | AI | 查 MCP，3 分钟摸清现有能力和依赖 |
| 09:31 | 业务调研 | AI | 查业务知识层：行业配置、AB 实验冲突、上下游字段 |
| 09:34 | 待确认项 | AI → 产品/研发 | 输出 AI 搞不定的业务判断题（3-5 个） |
| 09:36 | 审方案 | 研发 | 看 AI 方案，2 分钟调整 |
| 09:38 | 编码 | AI | 写代码 + 测试用例 + 提 PR，15 分钟 |
| 09:53 | 跑测试 | AI | 单测 + 预发部署 + 压测，5 分钟 |
| 09:58 | Review | 研发 | 审代码，提修改意见，10 分钟 |
| 10:08 | 改代码 | AI | 根据 review 修改，5 分钟 |
| 10:13 | 上线 | AI | CI/CD + 灰度 + 监控，5 分钟 |
| 10:20 | 完成 | - | 从需求到上线 50 分钟 |

**研发实际动手时间：12 分钟（审方案 2 分钟 + Review 10 分钟）。**

### 关键差异

| | 传统模式 | AI Ready 模式 |
|---|---|---|
| 调研依赖 | 人翻文档问同事，半天 | AI 查 MCP + 业务知识层，3 分钟 |
| 编码 | 人写，半天到一天 | AI 写，15 分钟 |
| 联调 | 人对着文档调，半天 | AI 直接调 MCP，不存在"文档过时" |
| 压测 | 提工单排队，1-2 天 | AI 调基建 MCP，5 分钟 |
| 上线 | 人操作，半小时 | AI 自动灰度 + 监控，5 分钟 |
| **总耗时** | **2-5 天** | **1 小时以内** |

## 七、业务知识层详解（核心难点）

传统模式下，业务逻辑散落在：产品的脑子里、代码的 if-else 里、历史聊天记录里。AI 读不到这些就是瞎子。

**AI 不是替代产品和研发的讨论，是把讨论范围从"所有事"缩小到"真正需要人判断的事"。**

### 需要结构化的内容

**行业配置矩阵**
```
行业: 美发
  - 优惠券: ✅ 支持，最大面额 50 元
  - AB实验: exp_1023(新客券面额测试) 进行中
  - 特殊规则: 节假日不发券

行业: 保洁  
  - 优惠券: ✅ 支持，最大面额 30 元
  - AB实验: 无
  - 特殊规则: 首单必须有券
```

**实验冲突规则**
```
同一用户同一行业不能同时命中两个券类实验
优先级: 留存实验 > 拉新实验 > 促活实验
冲突处理: 低优先级自动跳过
```

**上下游依赖关系**
```
批量发券 依赖:
  - B端商家服务: merchant.coupon_enabled 字段
  - 库存服务: GET /api/v1/coupon-template/{id}
  - 实验平台: GET /api/v1/ab/user-group
```

## 八、分工建议（7-8 人）

| 方向 | 人数 | 做什么 |
|---|---|---|
| **MCP 层** | 2-3 人 | 从下往上：基建 → 公共服务 → 业务 API，逐层包装成 MCP Server |
| **业务知识层** | 3-4 人 | **重头戏。** 结构化录入业务规则、行业配置、实验关系、上下游依赖 |
| **Skills + 验证** | 2 人 | 写编排逻辑，拿 AI Agent 跑真实场景，跑不通就知道差在哪 |

**从下往上建能力，从上往下定场景，中间碰头就通了。**

**最重要的一条：先选一个仓库打穿，别三个同时铺开。** 一个成功案例比三个半成品有说服力得多。

## 九、Skill 的定位

Skill 不是替代前端，**Skill 是替代"人脑中的操作流程"。**

- **没有 Skill**：客服遇到投诉 → 打开 4 个系统来回切换 → 20 分钟
- **有 Skill**：客服说"查一下这个用户的退款问题" → Skill 串联 4 个 MCP → 30 秒出结果

Skill 的边界：
- ✅ 能做：所有能通过 API 完成的事
- ✅ 能辅助：需要判断的事（AI 给方案，人拍板）
- ❌ 不能做：需要物理操作的事（但能做到"物理操作前的一切准备"）
