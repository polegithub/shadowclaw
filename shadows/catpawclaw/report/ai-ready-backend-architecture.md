# 后端架构 AI Ready 方案

> 2026-03-09

---

## 一、背景：为什么要做这件事

当前研发模式：产品提需求 → 研发理解需求 → 翻文档找依赖 → 写代码 → 联调 → 测试 → 部署。一个中等需求的交付周期通常在 2-5 天，其中真正写代码的时间可能只占 20%，大量时间花在沟通、调研、联调和等待上。

AI Agent 的能力已经到了可以独立完成"理解需求 → 编码 → 测试 → 部署"全链路的阶段。但前提是：**系统必须对 AI 友好**。现有架构是为人设计的——文档散落、接口不自描述、运维靠手动、业务规则藏在代码和人脑里。AI 面对这样的系统，和一个第一天入职却没有文档、没有导师的新人没有区别。

**这件事的本质：让我们的后端系统从"只有人能操作"变成"AI 也能操作"，从而将研发效率从天级提升到小时级。**

---

## 二、终态：我们要到达的地方

### 2.1 架构终态

```
人类（定义意图、审批、兜底）
       ↓
AI 编排层（理解意图 → 拆解任务 → 调度执行）
       ↓
Agent 服务层（标准化接口，所有能力原子化暴露）
       ↓
业务知识层（行业规则、实验配置、上下游依赖）
       ↓
服务层（业务 API + 公共服务，每个服务可被 AI 发现和调用）
       ↓
基础设施（DB / MQ / K8s，AI 可观测 + 可操作）
```

核心变化：**AI 成为系统的第一类用户。** 所有接口同时服务前端应用和 AI Agent，所有运维操作可被 AI 安全调用，所有业务规则 AI 可查询。

### 2.2 研发模式终态

| | 现在 | 终态 |
|---|---|---|
| **产品** | 写 PRD → 等排期 → 等开发 → 验收 | 描述需求 → AI 确认理解 → 几小时后验收 |
| **研发** | 接需求 → 写代码 → 联调 → 测试 → 部署 | 审方案 → 审代码 → 批上线 → 处理 AI 搞不定的 |
| **AI** | 不存在 | 需求拆解 → 调研 → 编码 → 测试 → 部署 → 运维 |

研发从"写代码的人"变成"审代码的人 + 建基础设施的人"。

### 2.3 用户界面终态

两种交互模式共存，共享同一套后端：

- **传统界面**：高频、确定性操作仍然点按钮（查列表、改状态、看报表）
- **AI 驱动**：非标准、跨系统、模糊意图的操作用自然语言完成

前端界面不会消失，但会越来越薄。大量"为了操作数据而专门开发的页面"会被 AI 能力替代，留下来的是"为了信息展示和用户体验必须存在的界面"。

## 三、如何达到：要做哪些事

### 3.1 能力暴露 — 让 AI 能"操作"系统

**目标：** 把现有系统的能力以标准化、可发现、可调用的方式暴露给 AI Agent。

**具体事项：**
- 每个核心服务（业务 API、公共服务、基建）封装为 **Agent 服务**（标准化的工具接口），AI Agent 可直接发现和调用
- 运维操作（部署、回滚、扩缩容、数据修复）封装成原子化命令，而不是藏在 wiki 里的手动步骤
- 操作分级：只读 / 可写 / 危险，危险操作需人工审批
- 基建能力（DB 查询、日志检索、监控指标、压测）同样暴露为 Agent 服务

**关键原则：** 接口协议会演进（今天的主流标准未来可能被替代），但"能力原子化 + 标准化暴露"的思路不会变。所以重点是把能力拆干净、描述清楚，而不是绑定某个特定协议。

### 3.2 业务知识结构化 — 让 AI 能"理解"业务

**目标：** 把散落在产品脑子里、代码 if-else 里、历史聊天记录里的业务逻辑，变成 AI 可查询的结构化知识。

**这是整件事最难也最关键的部分。** 技术能力暴露相对标准化，但业务知识是每个团队独有的壁垒。

**需要结构化的内容：**

**行业配置矩阵** — 哪些行业支持哪些功能、有什么特殊规则
```
行业: 美发
  - 优惠券: ✅ 支持，最大面额 50 元
  - 当前实验: exp_1023(新客券面额测试)
  - 特殊规则: 节假日不发券
```

**实验冲突规则** — 优先级、互斥关系、冲突处理策略
```
同一用户同一行业不能同时命中两个券类实验
优先级: 留存 > 拉新 > 促活
冲突处理: 低优先级自动跳过
```

**上下游依赖关系** — 接口字段、前置条件、数据准备状态
```
批量发券 依赖:
  - B端商家服务: merchant.coupon_enabled
  - 库存服务: GET /api/v1/coupon-template/{id}
  - 实验平台: GET /api/v1/ab/user-group
```

**AI 不是替代产品和研发的讨论，是把讨论范围从"所有事"缩小到"真正需要人判断的 3-5 个问题"。**

### 3.3 场景编排 — 让 AI 能"串联"流程

**目标：** 把人跨系统操作的 SOP 变成 AI 可执行的编排逻辑（Skills）。

Agent 服务解决"AI 能调到单个接口"，但业务场景往往需要串联多个服务。Skills 补的是这个：**什么时候调什么、怎么组合、出错了怎么处理。**

- 没有 Skills：客服遇到投诉 → 打开 4 个系统来回切换 → 20 分钟
- 有 Skills：客服说"查一下这个用户的退款问题" → AI 自动串联 4 个服务 → 30 秒出结果

**Skills 的边界：**
- ✅ 能做：所有能通过 API 完成的事
- ✅ 能辅助：需要判断的事（AI 给方案，人拍板）
- ❌ 不能做：需要物理操作的事（但能做到物理操作前的一切准备）

### 3.4 研发闭环 — 让 AI 能"交付"代码

**目标：** AI 能独立完成从编码到上线的全流程，人只做审批。

- CI/CD 流水线对 AI 开放（AI 提 PR → 跑测试 → 人审批合入）
- 测试覆盖率拉上去，AI 改完代码能自验证
- 代码规范、架构约束文档化，AI 写代码自动遵循
- 灰度发布 + 自动监控，AI 部署后自己盯异常

## 四、为什么这样可以达到：一个需求的全流程模拟

**需求：商家优惠券批量发放接口，5000 QPS，幂等，失败可重试。**

| 时间 | 阶段 | 谁做 | 做什么 |
|---|---|---|---|
| 09:30 | 需求输入 | 研发 | 一句话描述需求 |
| 09:31 | 技术调研 | AI | 查 Agent 服务：现有接口、依赖组件、DB 承载力，3 分钟 |
| 09:31 | 业务调研 | AI | 查业务知识层：行业配置差异、AB 实验冲突、上下游数据就绪情况 |
| 09:34 | 待确认 | AI → 产品/研发 | 输出 3-5 个 AI 搞不定的业务判断题 |
| 09:36 | 审方案 | 研发 | 看方案，2 分钟调整 |
| 09:38 | 编码 | AI | 写代码 + 28 个测试用例 + 提 PR，15 分钟 |
| 09:53 | 测试 | AI | 单测 + 预发部署 + 5000 QPS 压测，5 分钟 |
| 09:58 | Review | 研发 | 审代码，发现边界问题，提修改意见，10 分钟 |
| 10:08 | 修改 | AI | 根据 review 改代码 + 补测试，5 分钟 |
| 10:13 | 上线 | AI | 合并 → CI/CD → 灰度 10% → 监控 5 分钟 → 全量，5 分钟 |
| 10:20 | 完成 | — | **从需求到上线：50 分钟。研发动手：12 分钟。** |

### 为什么能这么快

| 传统瓶颈 | AI Ready 后 | 原因 |
|---|---|---|
| 调研半天 | 3 分钟 | AI 直接查 Agent 服务 + 业务知识层，不用翻文档问人 |
| 编码半天 | 15 分钟 | AI 写代码，且依赖接口都是自描述的，不用猜 |
| 联调半天 | 不存在 | AI 直接调 Agent 服务，不存在"文档过时" |
| 压测等 1-2 天 | 5 分钟 | 压测能力也是 Agent 服务，AI 自助调用 |
| 部署半小时 | 5 分钟 | CI/CD + 自动灰度 + 自动监控 |
| **合计 2-5 天** | **< 1 小时** | |

这不是理论推演——**当每一层能力都以标准化方式暴露、业务知识可被查询、研发流水线对 AI 开放时，这个流程中没有任何一步是 AI 做不到的。** 唯一需要人的环节是方案审批和代码 Review，这恰恰是人应该做的事。

## 五、要做的事清单

| # | 事项 | 说明 |
|---|---|---|
| 1 | 核心服务能力封装 | 业务 API、公共服务、基建逐层封装为 Agent 服务 |
| 2 | 业务知识结构化 | 行业配置、实验规则、上下游依赖录入可查询的知识库 |
| 3 | 场景 Skills 编写 | 核心业务流程编排为 AI 可执行的 Skills |
| 4 | 研发流水线开放 | CI/CD、压测、监控对 AI 开放，支持自助调用 |
| 5 | 安全护栏建设 | 操作分级、审批机制、审计日志 |
| 6 | 端到端验证 | 选一个仓库打穿全流程，验证模式可行后复制 |

**优先级：先选一个仓库打穿，一个成功案例比三个半成品有说服力得多。**
